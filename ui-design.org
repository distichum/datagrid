#+TITLE: Datagrid UI: Design Exploration
#+AUTHOR: Joshua Lambert
#+DATE: 2026-05-10

* Purpose

Datagrids currently have no UI aspect. This document explores how a UI
might be manifested in Emacs without committing to any implementation
yet. It surveys the rendering options Emacs offers, argues for vtable
as the most promising starting point, and records ideas worth
borrowing from =ess-view-data=, which is the closest existing analogue
to what we want to build.

- The UI must not be directly tied to datagrids because it should not
  depend on datagrid. Instead, this UI will take regular emacs data
  structures and present them in a uniform way.
  - Datagrid must work with this but that code can live in the
    datagrid package. This one will be different.

* Possible rendering approaches in Emacs

There are several built-in or near-built-in mechanisms that could host
a datagrid UI. They differ mainly in how much structure they impose on
the view layer and how much editing they support.

** tabulated-list-mode (built-in)

This is the engine behind =list-packages=, =proced=, and similar
buffers. It provides column headers, sortable columns, fixed-width
formatting, and a standard navigation keymap for free. It is the path
of least resistance and is well suited to read-only or
mark-and-act-on-row workflows in the style of Dired. Its main
limitation is that cells are not editable in place and very wide grids
force horizontal scrolling.

** tabulated-list-mode plus transient menus

The same view layer, but with =transient.el= layered on top to provide
a Magit-style discoverable command surface for filter, sort, group,
and mutate actions. This avoids writing a custom keymap UI and maps
cleanly onto the dplyr-style verbs the datagrid library already
exposes.

** org-mode tables and orgtbl-mode

If editing matters more than scale, an Org table is already a
spreadsheet-like UI with in-cell editing, formulas, and alignment.
A datagrid would be rendered into an Org table buffer and parsed
back. This is appealing for small grids but round-tripping types is
fiddly and Org tables fall over past a few thousand rows.

** ewoc (built-in)

Lower-level than tabulated-list-mode, but each row is bound to a live
object that can be mutated and redrawn individually. This is worth
considering if partial updates matter, for example recomputing one
column without redrawing the whole grid, or if non-rectangular row
presentations such as expandable master/detail rows are desired.

** vtable (built-in since Emacs 29)

A modern table library by Lars Ingebrigtsen. It supports
variable-pitch fonts, per-column formatters, and easier programmatic
refresh than tabulated-list-mode. If we can require Emacs 29 or
later, this is the strongest default today.

** SES (built-in) and ctable (third party)

=ses= provides actual in-cell editing and formulas but is not built
for large data. =ctable= predates vtable and is largely superseded.
Both are noted for completeness rather than as serious candidates.

* Cross-cutting features to lean on

Regardless of the rendering choice, several built-in facilities are
likely to be useful:

- =text-properties= and =overlays= for per-cell faces, type hints,
  and NA highlighting.
- =revert-buffer-function= as a natural hook for reloading the
  underlying data.
- =imenu= for jumping to grouped sections in a grouped or summarized
  view.
- =header-line-format= for sticky column headers when scrolling.
- =display-buffer-alist= so users can choose side-window versus
  popup placement.
- =bookmark.el= for saving a grid plus filter state as a bookmark.
- =transient= for verb dispatch.
- =edit-indirect= or the built-in =clone-indirect-buffer= for
  structural edits to filter or mutate expressions.

* Why vtable looks like the right choice

Among the options above, vtable strikes the best balance for the
datagrid use case for several reasons.

It is built in as of Emacs 29, so we incur no external dependency. It
exposes per-column =:getter= functions, which means the buffer never
needs to own the data; the datagrid object remains the source of
truth and the view simply asks for the values it needs. It supports
=vtable-update-object= for redrawing a single row, which makes
edit-then-commit cycles cheap. It also supports per-column and
per-row keymaps, so binding a key like =e= on a cell to a
type-appropriate reader is straightforward.

What vtable does not do is in-place cell editing. A user cannot land
the cursor in a cell and type. This is acceptable, and arguably
preferable, for a typed-column grid: editing through the minibuffer
lets us dispatch to the appropriate reader for each column type, for
example =read-number= for numeric columns, =org-read-date= for
dates, and =completing-read= over factor levels for categorical
columns. Invalid input can be rejected before it ever touches the
data.

* Handling large datagrids

Datagrids are intended to be possibly very large, which exceeds what
any tabular UI in Emacs is comfortable rendering directly. The
solution is to ensure the UI is always a /view/ over a small slice of
the data, never the data itself. Several strategies, in increasing
complexity, are available.

** Window or page model

Maintain a =(start, size)= window into the datagrid. The vtable only
ever sees =size= rows, typically two hundred to one thousand. Keys
shift the window and the table is rebuilt. This is cheap, predictable,
and the user always knows where they are in the grid.

** Virtual indices via getter indirection

The =:objects= list given to vtable does not have to be the real
rows. It can be lightweight handles, for example integer indices, with
each column's =:getter= looking the value up in the datagrid on
demand. Combined with the window model, the swap between pages
becomes a change to the index list rather than any data copying.

** Filter first, then view

Compose a =filter=, =arrange=, or =select= pipeline through transient
menus, materialize the much smaller result, and hand only that to
vtable. This matches how people actually work with large tables: you
almost never want to look at two million rows; you want to look at
the eight hundred that matter.

** Chunked lazy loading

Detect when point reaches the last row and append the next chunk via
=vtable-insert-object=. This feels seamless but adds complexity around
sorting partially loaded data and is not recommended unless the
simpler approaches prove insufficient.

The recommended starting point is to combine filter-first with a
modest window. The filter pipeline is the primary navigation
mechanism, and the window is the fallback for users who genuinely
want to scroll raw rows. The =swap= between views is then nothing
more than rerunning the pipeline, getting a new result grid, handing
its rows to vtable, and reverting the table.

* Ideas worth borrowing from ess-view-data

=ess-view-data= is the closest existing analogue to what we are
contemplating: it presents R data frames in Emacs, supports dplyr-style
verbs, and handles the size problem. Reading the source revealed
several design choices worth borrowing.

** Pagination is the default size strategy

The default =ess-view-data-rows-per-page= is two hundred, and the DT
backend uses one thousand. The maintainers did not try to render
whole frames; they paginated from the start. This validates the
window-or-page approach above.

** The action history is itself an editable artifact

The pipeline of operations the user has applied is treated as code
that can be revised, not as opaque UI state. This is a powerful idea
for our datagrid as well. A user who built up a chain of filters and
mutates should be able to open the chain, edit a step, and re-run.

** Structural edits happen in indirect buffers

Cell-level edits are a fine fit for the minibuffer, but
=ess-view-data= makes a strong case that /structural/ edits, meaning
filter predicates, mutate expressions, and the pipeline itself, want
a real buffer. Multi-line input, syntax highlighting, completion, and
an explicit commit step all matter here. The package opens an
=edit-indirect= buffer holding just the expression, lets the user
edit it as code, and commits on =C-c C-c=. This suggests a two-tier
edit model for the datagrid UI:

- The minibuffer for changing a single cell value.
- An indirect buffer for editing a filter expression or a pipeline
  step.

** A backend abstraction keeps the UI uncoupled from the engine

=ess-view-data-current-backend= defaults to =dplyr= but the same UI
can drive other engines. Our datagrid already speaks a dplyr-shaped
verb vocabulary, so the same abstraction is essentially free, and it
leaves room for alternative implementations later.

** Expressions are the UI, not widgets

Rather than wrapping each filter clause in a form widget,
=ess-view-data= lets the user write expressions directly and edits
them as text. This is more idiomatic in Emacs and avoids the trap of
building a constrained GUI that cannot express what users actually
want.

* What ess-view-data does not do

Two things are worth noting because they shape what we should /not/
copy.

It does not use =tabulated-list-mode=. =ess-view-data-mode= is a
minor mode layered on top of whatever buffer R's print output landed
in. The view is essentially formatted text with a keymap and a
header line. This works for them because R does the formatting, but
it would not work for us; we want a real table widget, which is
exactly what vtable gives us.

It does not edit cells in place or via the minibuffer. All edits go
through an indirect buffer, even for single values. For a typed
datagrid, the minibuffer plus type-aware reader path is a better fit
for cell edits, and the indirect buffer should be reserved for
structural edits where multi-line code is genuinely useful.

* Tentative direction

Pulling these threads together, the design that emerges is:

- vtable as the rendering layer.
- A page or window plus a filter pipeline as the size strategy, with
  the pipeline being the primary navigation surface.
- Transient menus for verb dispatch.
- Minibuffer with type-aware readers for cell edits.
- Indirect buffers for structural edits to expressions or pipeline
  steps.
- The pipeline itself stored as an editable artifact, in the spirit
  of =ess-view-data='s action history.
- The datagrid object remains the source of truth; the buffer is
  always a view.

No code is written yet. This document captures the shape of the
problem and the candidate solution so the next round of thinking can
build on it.
