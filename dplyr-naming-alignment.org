#+TITLE: dplyr Naming Alignment for datagrid
#+DATE: 2026-05-07

Comparison of datagrid's public function names against dplyr's verb set.
Items below are candidates for rename (with deprecated aliases) or
addition. The =datagrid-join= family has already been migrated to
dplyr-style names; this list covers the rest of the surface.

* Overarching principles
Mutating datagrids in place is generally discouraged. Unless a function
is specifically for mutating, don't do it.

** Examine all datagrid code to see where mutations happen
Prompt the user for decisions.

* Real misses (function exists, name diverges)

** DONE datagrid-column-frequencies → datagrid-count
:PROPERTIES:
:PRIORITY: A
:END:
Same operation as dplyr =count=. Rename and keep
=datagrid-column-frequencies= as a deprecated alias.

Signature:
#+begin_src elisp
(datagrid-count DG COL &key sort name)
#+end_src
The required COL is what disambiguates this from "row count" or
"column count" — =datagrid-count= always operates on a column's
values. =:sort= orders by frequency; =:name= sets the count column's
heading (default "n").

*** Further instruction

** DONE datagrid-column-unique → datagrid-distinct
:PROPERTIES:
:PRIORITY: A
:END:
dplyr (and tibble) call this =distinct=. Note: the existing function
operates on a single column. Decide whether =datagrid-distinct= should
also support a row-level form (drop duplicate rows across all or
several columns), which is dplyr's primary semantics. Likely:
- =datagrid-distinct= (rows) — new, multi-column row dedup
- =datagrid-distinct-column= or just keep the single-column case under
  =datagrid-distinct= with an index argument

Signature (folded form):
#+begin_src elisp
(datagrid-distinct DG &rest COLS)
#+end_src
- No COLS → row-level dedup across all columns (dplyr's primary
  semantics).
- One COL → unique values from that column.
- Multiple COLS → dedup rows considering only those columns.
*** Further instruction
Support row-level form but document its performance problems.

** DONE datagrid-reduce-vec / datagrid-reduce-vec-calc → datagrid-summarize
:PROPERTIES:
:PRIORITY: A
:END:
=reduce= is Lisp/Clojure flavor; data-analysis users reach for
=summarize= / =summarize=. Possible split:
- =datagrid-summarize-column= for the simple per-column reduction
- =datagrid-summarize-column-calc= for the calc-wrapped variant
Or fold both into =datagrid-summarize= with a flag.

Signatures (split form preferred — second arg has a different type
in each):
#+begin_src elisp
(datagrid-summarize-column      DG COL FN        &key name)
(datagrid-summarize-column-calc DG COL CALC-EXPR &key name)
#+end_src
Folded alternative:
#+begin_src elisp
(datagrid-summarize DG COL REDUCER &key calc name)
#+end_src
where =:calc t= reinterprets REDUCER as a calc expression string.

*** Further instruction
Do not change datagrid-reduce-vec or datagrid-reduce-vec-calc.
Instead, create a similar datagrid-summarize-column. Reuse code. Use
summarize, not summarize. Use the following form. Uniform, parses with
plist-get, no ambiguity. A bit more verbose but every field is
labeled.

Plist spec (keyword-only, no positionals):

  #+begin_src emacs-lisp
  (datagrid-summarize
   sales-dg
   '(:name "total_revenue" :fn + :col 2 :convert t)
   '(:name "max_units"     :fn max :col 3 :convert t)
   '(:name "top_region"    :fn datagrid-mode :col 1 :code t))
  #+end_src

** DONE datagrid-get-col-data / datagrid-col-data-by-header → datagrid-pull
:PROPERTIES:
:PRIORITY: A
:END:
Both functions extract a column's data as a vector — exactly dplyr
=pull=. Collapse to a single =datagrid-pull= that accepts either an
integer index or a heading string (use the same resolver pattern as
=datagrid-join--resolve-key=).

Signature:
#+begin_src elisp
(datagrid-pull DG COL)
#+end_src
COL is an integer index or a heading string. Returns a vector.

*** Further instruction

** WONTDO datagrid-sort → datagrid-arrange (weak case)
:PROPERTIES:
:PRIORITY: C
:END:
dplyr uses =arrange=, but base R and most languages use =sort=.
Probably leave alone unless going strictly dplyr-faithful. Listed for
completeness.

Signature (if added):
#+begin_src elisp
(datagrid-arrange DG &rest COLS)            ; ascending
(datagrid-arrange DG SPEC)                  ; SPEC alist: ((col . 'desc) ...)
#+end_src
The alist form handles per-column direction more cleanly than parallel
lists.

*** Further instruction
Leave alone.

* Naming gaps (no function exists, but expected by name)
** DONE datagrid-rename
:PROPERTIES:
:PRIORITY: A
:END:
Single-column heading rename. Today only =datagrid-set-headings= (all
columns at once) exists. Should accept an index or current heading and
a new heading; consider also a multi-rename form taking an alist.

Signature:
#+begin_src elisp
(datagrid-rename DG COL NEW-HEADING)   ; single
(datagrid-rename DG RENAMES)           ; alist: ((old-or-idx . new) ...)
#+end_src
Dispatch on whether the second arg is an atom or an alist.

*** Further instruction

** DONE datagrid-select
:PROPERTIES:
:PRIORITY: A
:END:
Pick a subset of columns by index list or heading list, returning a
new datagrid. =datagrid-remove-column= exists for the negative form;
no positive form yet.

Signature:
#+begin_src elisp
(datagrid-select DG &rest COLS)
#+end_src
COLS may mix integer indices and heading strings, resolved via the
shared resolver.

*** Further instruction

** DONE datagrid-mutate
:PROPERTIES:
:PRIORITY: A
:END:
Apply a function across one or more existing columns to produce a new
column (or replace one). Today users hand-build a
=datagrid-column-make= and call =datagrid-add-column=. A
=datagrid-mutate= helper that takes a heading, a function, and one or
more source-column indices would be a major ergonomic win.

Signature:
#+begin_src elisp
(datagrid-mutate DG NEW-HEADING FN &rest SOURCE-COLS)
#+end_src
FN is applied row-wise, receiving one value per SOURCE-COL. If
NEW-HEADING matches an existing column it is replaced; otherwise the
column is appended.

*** Further instruction

** DONE datagrid-slice
:PROPERTIES:
:PRIORITY: B
:END:
Pick rows by an index list. Already exists privately as
=datagrid--select-rows=; promote to public and rename.

Signature:
#+begin_src elisp
(datagrid-slice DG &rest ROW-INDICES)
#+end_src
Mirrors =datagrid-select= but along the row axis.

* Things that look off but aren't

- =datagrid-from-*= / =datagrid-to-*= — I/O constructors. dplyr
  doesn't dictate this; the from/to naming is clearer than =as-=.
- =datagrid-create-mask= / =datagrid-filter-by-mask= — different
  abstraction (numpy-style two-step filter), not a naming miss. Could
  optionally add a =datagrid-filter= that takes a predicate directly,
  layered on top of the existing primitives.
- =datagrid-add-column= / =datagrid-add-row= — dplyr would call these
  =bind_cols= / =bind_rows=, but =add-= reads fine in Lisp.
- =datagrid-head= — already matches dplyr / R.
- =datagrid-group-by= — already matches.
- =datagrid-rows-patch= — keyed cross-table NA-fill, matching dplyr
  =rows_patch=. Renamed from =datagrid-coalesce-column(s)= 2026-05-07
  because that name implied row-wise coalesce semantics, which it does
  not have. For row-wise coalesce of overlapping columns after a join,
  compose =datagrid-mutate= with `or'.

* Suggested order of operations

If migrating in stages:

1. Pure renames first — they're mechanical and low-risk:
   =count=, =distinct= (single-col), =summarize=, =pull=. Each gets a
   deprecated alias following the =datagrid-join= → =datagrid-left-join=
   pattern.
2. New verbs next — =rename=, =select=, =mutate=, =slice=. These
   shape the API the most for new users.
3. Optional: =arrange= alias for =sort=, =filter= sugar over the
   mask functions, row-level =distinct=.

* Migration pattern reference

Match the pattern used for joins:

#+begin_src elisp
(defun OLD-NAME (...)
  "Deprecated. Use NEW-NAME instead."
  (declare (obsolete NEW-NAME "1.0"))
  (NEW-NAME ...))
#+end_src

Tests written against OLD-NAME keep passing; new code uses NEW-NAME.
